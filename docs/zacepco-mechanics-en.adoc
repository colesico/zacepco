== About this Document

This document describes the mechanics of a detective game, establishing the general rules by which it must be constructed.
For the integrity of the game, an integral addition to these mechanics are the *crime scenarios*, which describe the narrative component.
The mechanics themselves are essentially a *framework* for a deductive detective game with elements of randomness.

== Crime Scenario

A scenario within the mechanics describes a crime that has already occurred.
During the game, players (detectives) must solve this crime.
The scenario caters to varying degrees of difficulty in solving the crime, describing, for example, a single crime or multiple at once, involving one or several criminals, with or without accomplices, with different numbers of suspects, the duration of the crime, the size of the "theater of operations", etc.

Any scenario contains:

* Instructions for preparation before the investigation begins.
* Information for the course of the investigation.
* An "Answers" section, which can be viewed at the end of the game.

Ready-made scenarios come with the game or are generated by a special program.

The game is played according to the scenario chosen by the players and is divided into two *phases*.

In the first phase, players perform preparatory actions according to the scenario to start the investigation: assembling the game board, etc.

The second phase is the investigation, a cooperative process of solving the crime by the detectives.

At the end of the game, the conclusions of the investigation are compared with the "Answers" section of the scenario.

== Master Program

A computer program — the *"Master Program"* — is used to moderate the game process. It is configured by the scenario for a specific game.
The program also automates all calculations related to the game process and controls the game mechanics.

== Game Entities and Relationships

The scenario operates with the following types of *game entities*:

* *Character ( P )* — a person involved in the crime;
* *Location ( L )* — a part of the crime scene;
* *Clue ( C )* — a piece of information about the crime;
* *Time ( T )* — a moment in time during the crime.

The game entities themselves (except for Time) are represented by entity cards.
Each such card, depending on the entity type, has its own format.
An entity is identified by a unique identifier consisting of a letter - the entity type - and a unique number. For example, P2, L5, C55.

Time does not have its own cards, only a serial number (T1, T2, …).

Each game entity can be linked by the scenario to any other entities.
From the perspective of a relationship, when one entity references another, the first is called the *key entity*, and the second is the *target entity*.

The scenario defines a set of game entities and their relationships, which essentially describes the crime (defines the crime model).

Some of these entities and relationships may not be related to the crime and serve to complicate the investigation process (false clues).

Before the investigation begins, the detectives know a small part of the entities and their relationships.
During the investigation, the detectives gradually uncover the remaining entities and relationships.

In the game, entities are represented by cards. Some of the entity cards used in the scenario are pre-made — *"reference entities"* — and some require fabrication — *"custom entities"*. They are provided as templates within the scenario and need to be printed by the players before the game. Reference entities can be reused in different scenarios; the scenario defines custom entities when necessary.

== Uncovering Entities and Relationships

The game uses a mechanic of *step-by-step* uncovering of entity cards by the detectives and discovering relationships between them.

Based on some already uncovered key entities, detectives can learn their relationships with other target entities, both uncovered and not yet uncovered.
Then they can choose other key entities and again determine subsequent target entities and relationships, and so on.

Relationships between entities, depending on the specific game mechanic, are described by the scenario in the form of *relationship tables* of a specific format.

When obtaining information from the relationship tables, the *master program* is used to avoid revealing all information about the crime to the detectives at once.

== Randomness Element

D12 die.

== Resource, Points

In the game, each detective action costs a certain *resource*.
If an action is successful, the player is awarded *points*.
The scenario defines the total *budget* of resources for the investigation.
When the budget is exhausted, the investigation stops, and the detectives must draw conclusions based on the available data and compare the results with the answers.

The individual number of points earned determines the best detective in the investigation.

== Game Time

The crime scenario covers a certain period of time.
Time within this period flows discretely, in *ticks*, from first to last (T1, T2,...Tn).
The crime lasts for a number of ticks specified in the scenario.
The more ticks, the more difficult it is to solve the crime.
All events in the crime are tied to ticks.
Something happens in every tick of time.
It is assumed that nothing happens between ticks.

== Crime Scene, Locations, Transitions

In the scenario, the crime scene is defined as a graph.
In the preparation phase for the investigation, players assemble the crime scene — the stage — based on this graph from individual *locations* (graph nodes). The stage is a grid of a specific size (the grid size is set by the scenario). A location is placed in any cell of the stage grid. Some cells of the stage may be left empty.

Each location is represented by a square location card (reference or custom). The location card contains the following information:

* ID;
* Name;
* Description;
* Location image.

Adjacent locations can be connected by *transitions* for a character.
If it is necessary to connect non-adjacent locations with a transition, transition cards with different configurations (straight, turn, T-shaped) are used, which are placed in empty stage cells, connecting the required locations.
Nothing ever happens in the transition itself; it merely serves to indicate the possibility of character movement between locations.
If it is necessary to model character behavior in a transition (e.g., travel time, some actions), it is replaced by a full-fledged location.

The scenario may also assume the presence of *"hidden" transitions* between locations, which are not indicated during the preparation phase for the investigation.
Detectives learn about the existence of such transitions during the investigation.
When a clue indicating the presence of a hidden transition (*hidden transition clue*) is uncovered in a location, the ID of the target location to which the transition leads is determined.
A transition card is placed on the stage, connecting the location where the hidden transition clue was uncovered with the target location.

A hidden transition may lead to a *hidden location*.
This is a location that the detectives do not know about before the investigation begins.
One of the *reference* locations not yet placed on the stage serves as the hidden location.
A discovered hidden location is placed on the stage in the cell specified in the scenario.
The location is connected by a transition to the location where the hidden transition clue was found.
After being uncovered, the location becomes available for investigative actions.

Information about all hidden transitions and locations is included in the scenario *answers*.

When developing a scenario, locations must be defined in such a way that their meaning in the investigation is taken into account.
For example, a cave may have no light, a character may only enter a basement with a key, a poisonous plant may grow in a greenhouse, a bar may be open from tick 3 to tick 6, etc.
The semantic load and description of the location should help detectives in the investigation.

The scenario describes an event log for each location for each time tick.
The log event contains information about which characters were in this location, which clues they left or took from the location (clues that interacted with the location).

The event log is used at the end of the game to verify the conclusions of the investigation and is not available to players during the investigation.

== Characters

An acting person in the crime is a *character*.
Each character is represented by a (reference or custom) character card.
The character card contains the following information:

* ID;
* Name;
* Dossier (description);
* Character image.

The set of characters in the game is determined by the scenario.
The scenario defines explicit characters — they are known before the investigation begins.
Their cards are placed next to the game board.

The scenario may also imply the presence of *hidden characters*.
A hidden character is one whose card has not yet been placed face-up on the table from the character deck.
One of the *reference* characters serves as the hidden character.

The presence of a hidden character can be determined during the investigation logically, based on discovered clues, or when a clue with a direct reference to this hidden character is uncovered (similar to clues indicating a hidden transition).
When a clue referencing a hidden character is found, their card is placed face-up from the character deck next to the cards of the other characters on the game board.
After this, the character becomes available for investigative actions.

In the scenario, in the "Answers" section, all hidden characters are listed.

Characters in the scenario should be described in such a way that this description helps detectives in the investigation.

== Character Movements and Actions

Each character during the crime *in each tick* of time has only the following possibilities:

* Move to an adjacent location or stay in the current one.
* While in a location, perform *one* action or do nothing.

Movement occurs to an adjacent location if a transition exists.
On the scenario side, some restrictions on the transition are possible, for example, a door that needs to be opened with a key.
In general, a character can visit the same location multiple times in different ticks.
For easier game variants, a character may visit a location only once.

The character's movement path (sequence of locations) is implied by the scenario but is not explicitly revealed to the players.
Players can determine this path during the investigation.

Character paths are revealed by the scenario in the "Answers" section: for each character, their movements are indicated as a sequence of location IDs for each tick (in the character's log).
This information can be viewed at the end of the game.

During the investigation phase, to visualize the path (*track*) of a character's movement across the game board, tokens of the character's color are placed in locations.
On each token is the tick number during which the character was in the location where the token is placed.
Such tokens are called character track tokens.

While in a location, a character can perform *one* action.
The mechanics do not define the action itself as an entity, but the result of this action becomes a *clue* that is linked either to a location or to a character.

The scenario describes an event log for each character for each time tick.
The log event contains information about which location the character was in and which clue they interacted with (e.g., left a trace or a trace remained on the character).

The event log is used at the end of the game to verify the conclusions of the investigation and is not available to players during the investigation.

== Clues

A *clue* is a piece of information about the committed crime that detectives sequentially discover during the investigation.
A clue can be any physical object, event, some information, or fact.

Examples of clues: witness testimony, motive, a note, a cigarette butt, a habit, a character's addiction to smoking, the fact of an object's absence from a location, a laboratory report.

The crime scenario defines the set of clues involved in the game.
A clue is represented by a (reference or custom) clue card printed with:

* ID;
* Name;
* Description (detailed);
* Clue image.

Some clues may not be related to the crime and serve to make the game more difficult (false clues). The more such clues, the more difficult the game.

Initially, clues are hidden from the detectives.
During the investigation, detectives gradually "uncover" clues — placing clue cards face-up.

If an uncovered clue contains references to other not-yet-uncovered entities, those entities are also uncovered (recursively). The scenario should not define deep recursion of uncovered entities.

Examples of clues:

* Clue C1: "Testimony. Saw character P1 (Maid) during tick 4 in location L2 (Dining Room)."
* Clue C2: "A hidden passage to location L4 (Basement) was found in this location."
* Clue C6: "C2 (Candlestick) is missing from this location, but was originally here."
* Clue C12: "Broken clock. The hand points to T8 ticks."
* Clue C13: "Character P1 (Gardener) was seen in this location at T3 tick with C6 (Knife) in hand."

== Variable Values

To universalize printed entity cards, instead of specific values/text, variable names are indicated, the values of which are determined in the scenario.

Example of using a variable for a clue:

====
C1: Saw character $P during $T tick in location $L
====

The values of P, T, L are taken from the scenario.

====
L2: Location $N. Description: $D
====

The name N and description D are taken from the scenario.

== Entity Reuse, Reference and Custom Entities

A scenario may use some entities from other scenarios.
To do this, the scenario specifies from which scenarios the entities will be used. For these "imported" entities, "local" entities are created (like regular scenario entities), in which a reference to the imported entity is indicated.
Such entities are called *reference entities*.

All scenario entities that do not have references to imported entities are considered *custom entities* (local scenario entities) and are printed before the game starts.

== Searching a Location

During the investigation, detectives can *"search"* any uncovered location (including previously hidden ones) to try to uncover clues associated with that location.
As a result of the search, a clue may be "discovered" (uncovered) with a certain probability.

The scenario defines a relationship table *"Location Search"*, which defines the relationship between locations and clues.
During a search, a mechanic of probabilistic clue discovery is used together with the *"Search Focus"* mechanic.
The focus allows for a higher probability of discovering the clue on which the search focus is directed and to gain more points.
However, the strategic goal of the focus is rather not to "get" the clue, but to *formulate a hypothesis for testing*.
The focus consists of two parts: a tabular focus (reference) and a detective's focus.

Searching a location costs a certain amount of resources. The detective may be awarded points for the search.

Search Procedure:

1. The detective announces a search in a specific location, e.g., L1.
2. Specifies the detective's focus — a set of IDs of any *uncovered* entities or any types, e.g., T4, L?, C?.
3. Rolls a D12 die to model the probabilistic nature of clue discovery.
4. The result of the search is determined from the "Location Search" table.
   Access to the search table is via the master program.

=== Relationship Table "Location Search"

|===
|Location |Focus            |Threshold|Clue
| L1      | P4, L3, L4, T4  |5.2      |C8
|  *      | L4, C5, C6      |5.0      |C3
| L5      |     L6          |8.0      |C12
|===

*Location* — ID of the location being searched.

[[search-focus]]
*Focus* — tabular search focus. Serves to increase the probability of uncovering clues that have relationships with the target entities of interest to the detective.
The tabular focus lists only the _IDs_ of entities related to the target clue.
Specifying individual types in the tabular focus is not allowed.
You can list several IDs of the same type separated by commas — this is interpreted as any of them.

The detective's focus may consist of one or more IDs of *uncovered* entities or any entity types.
For example: "P1, C1, T?". The maximum number of elements in the detective's focus is determined by the scenario (usually 3).

*Threshold* — the threshold for discovering the clue. A _real_ number that is compared with the number rolled on the D12 die.
If the roll is less than the "Threshold", it is considered that negative factors played a role and the target clue is not discovered.
In the scenario, the discovery threshold is set so that the most important clues for the investigation have higher threshold values. The possibility of specifying fractional values allows for "fine" ordering of clues.

To reduce the influence of randomness, low threshold values should be specified; to eliminate randomness — less than 1.

*Clue* — ID of the target clue. The card of the discovered clue is placed face-up. To indicate the place where the clue was found, a token with the uncovered clue's ID is placed on the location.

=== Search Cost, Awarding Points

[[search-cost]]
*Search Cost* — the resource cost of searching is the same for any location and is determined in the scenario.
Determines how many resources are spent for a search in a location. The detective learns the cost at the beginning of the game.

[[search-points]]
*Points for Search* — the base number of points that may be awarded to a detective for searching a location.
The same base number of points is determined for all locations in the scenario.
Base points are awarded for uncovering a clue, but only if it was _NOT_ *<<accidental-discovery, accidentally discovered>>*.
When awarding base points, one additional point is awarded for each _entity_ match in the detective's focus with the tabular one.

== Algorithm for Determining the Target Clue in the Location Search Table

Goal of the algorithm: to determine the clue that best matches the detective's focus and the D12 roll.

Only rows of the search table related to the selected location are considered in the algorithm.
Rows for which the clue has already been uncovered in previous searches are excluded from consideration.

1. For each row, the focus weights EQ, EQT, NE are calculated (see <<focus-weights-comp, calculation of focus match weights>>).
2. The table rows are ordered by descending "EQ", descending "EQT", ascending "NE", descending *Threshold*.
3. Sequentially, starting from the first row, the "Threshold" is compared with the number on the D12 die.
4. If the D12 value >= "Threshold", the scan stops. The clue is discovered, and it is uncovered.
   It is reported how many points are awarded to the detective.

[start=5]
[[accidental-discovery]]
5. If no suitable row is found (the D12 value is insufficient), an *"Accidental Discovery"* of a clue occurs.
   The clue with the lowest threshold among those considered (not yet uncovered clues) is uncovered.
   It is reported that a suspicious clue was accidentally discovered.
6. Matching elements (entities + types) of the _detective's focus_ with the tabular focus of the uncovered clue are reported:
   * Matched entity IDs and types in the detective's focus;
   * Types of unmatched entity IDs in the tabular focus.

====
For example, for the detective's focus "P2, T4, C?" and the tabular focus "P1, P2, T4, L2", the following will be output:
"In location L1, clue C2 was found, related to P2, T4, P?, L?".
====

[start=7]
7. If all clues of the location are uncovered, a "All clues discovered" token is placed on the location. Further searches of the location are impossible.

[[focus-weights-comp]]
=== Calculation of Focus Match Weights

The following focus match weight values are calculated:

* EQ = [number of matching entity IDs in the focuses],
* EQT = [number of matching types in the focuses],
* NE = [number of *NOT* matching entities or types] in the detective's search focus and the tabular focus.

First, "EQ" is calculated by matching IDs in both focuses. Then the matched IDs are removed from both focuses.
To calculate "EQT", the entity types explicitly specified in the detective's focus (P?, L?, …) are compared with the types of the remaining IDs in the tabular focus. Elements matched during the "EQT" calculation are removed from both focuses.
"NE" is calculated as the total number of remaining entities in the focuses.

When calculating "EQT", an element of type "X?" in the detective's focus is considered to match an entity in the tabular focus (with an ID) if the type "X?" corresponds to the type in the entity's ID.

For example, the detective's focus P1, P2, T? compared with the tabular P2, T4, C1 gives:
EQ = 1 (for P2), EQT = 1 (for T?), NE = 2 (for P1, C1).

== Interrogating a Character

The scenario links some clues with each character.
It is considered that part of them was with the character before the crime (e.g., clothing, personal belongings, character).
Part of them they acquired during the crime — picked up in locations or received from other characters (e.g., a knife, a note).
Part of the character's clues describe their inherent traits (e.g., habits, addiction to alcohol, character, etc.). The rest of the clues are the character's testimonies.
The scenario also assumes that during the crime, characters can leave clues in locations, transfer or plant them on other characters when meeting them in the same location.

Part of a character's clues may be known to the detectives (e.g., clothing, etc.) before the investigation.
Another part of the clues is hidden.

Similar to the location search procedure, to uncover such clues related to a character, a detective can *"interrogate"* any uncovered character (including previously hidden ones).
As a result of the interrogation, a new clue of the character may be uncovered and a connection with other clues established.

During the interrogation, the detective "asks" the character about facts known to the investigation (uncovered key entities) or about hypotheses of the investigation (about any entity types) — that is, they define the *interrogation focus*.
If the character "knows" something about this, a clue related to the focus and its relationships are uncovered.
The interrogation mechanic is designed such that without knowledge of key entities or their types, it is difficult for the detective to uncover the relevant target clue of the character.

In response, the character may *"lie"* — a clue is uncovered that is actually not related to the key entities in the interrogation focus.
The scenario regulates the number of "false" clues. The falsity of a clue can only be established through deductive analysis.
The "Answers" section lists which clues were indeed false.

Each character, to some extent, is unwilling to share certain information, and the detective uses psychological pressure to obtain maximum information.
This interaction has a random nature and is modeled using a D12 roll.

Interrogating a character costs a certain resource. The detective may be awarded points for the interrogation.

The scenario defines a relationship table *"Character Interrogation"*, which defines the relationship between characters and clues.

Interrogation Procedure:

1. The detective selects a character to interrogate, e.g., P1.
2. Specifies the detective's focus — a set of IDs of any *uncovered* entities or any types, e.g., L1, P?.
3. Rolls a D12 to model psychological pressure on the character.
4. The result of the interrogation is determined from the interrogation table.
   Access to the interrogation table is via the master program.

=== Relationship Table "Character Interrogation"

|===
|Character |Focus              |Threshold |Clue
|P1        |P2, L3, C4, C5, T2 | 3.0      |C3
|P4        |C4, T5, T6         | 5.0      |C12
| *        |P5                 | 5.2      |C15
|===

*Character* — the character being interrogated.
The detective, based on clues known from the investigation and logical conclusions, selects the character to interrogate.

*Focus* — tabular interrogation focus (same as <<search-focus, tabular search focus>>).

The detective's focus has a restriction:
*Among the elements of the detective's focus, there must be at least one entity ID.*

*Threshold* — the threshold of resistance to the detective's psychological pressure.
A _real_ number that is compared with the number rolled on the D12 die.
If the roll is less than the "Threshold", it is considered that the detective could not overcome the character's reluctance to share information, and therefore the target clue is not uncovered.
In the scenario, the discovery threshold is set so that the most important clues for the investigation have higher threshold values. The possibility of specifying fractional values allows for "fine" ordering of clues.

To reduce the influence of randomness, low threshold values should be specified; to eliminate randomness — less than 1.

*Clue* — the character's target clue, which is uncovered upon a successful interrogation.
The clue may be deliberately set incorrectly by the scenario — simulating the character lying / forgetting.

=== Interrogation Cost, Awarded Points

*Interrogation Cost* — the resource cost of interrogating any character is the same and is determined in the scenario.
Determines how many resources are spent for interrogating a character. The detective learns the cost at the beginning of the game.

*Points for Interrogation* — the base number of points that may be awarded to a detective for interrogating a character.
The same base number of points is determined for all characters in the scenario.
Base points are awarded for uncovering the target clue during interrogation (whether false or not).
When awarding base points, one additional point is awarded for each _entity_ match in the focus (regardless of whether the clue is uncovered).

[[interrogation-alg]]
=== Algorithm for Determining the Target Clue in the Character Interrogation Table

The goal of the algorithm for determining the target clue is: to determine the character's clue that best matches the detective's focus, but clues must be uncovered starting from the least significant for the investigation.

Rows of the interrogation table related to the selected character are considered.
Rows for which the clue has already been uncovered in previous interrogations are excluded from consideration.

1. For each row, the focus weights EQ, EQT, NE are calculated (see <<focus-weights-comp, calculation of focus match weights>>).
2. The row with the maximum "EQ", maximum "EQT", minimum "NE", and minimum *Threshold* is found.
3. If for this row "EQ = 0", the character does not know what the detective is asking about. The target clue is not uncovered.
   It is reported that the character does not have the required information:
   * Unmatched entity IDs and types from the detective's focus.
   * Entity types of unmatched IDs in the tabular focus (as a hint about other relationships).
====
For example, for the detective's focus "L3, P?, T?" and the tabular focus "L4, T1, C2", the following will be output:
"Character P1 does not know about L3, P?, but hints that they know about L?, T?, C?".
====

[start=4]
4. If "EQ > 0", the character knows what the detective is asking about.
   The following are selected for reporting:
   * Matched entity IDs and types in the detective's focus.
   * Entity types of unmatched IDs in the tabular focus.

5. If "D12 >= Threshold", the target clue is uncovered. The uncovering of the clue and its relationships are reported:

====
For example, for the target clue C8, the detective's focus "L3, P?, C?" and the tabular focus "L3, P7, P4, T2", the following will be output:

"Character P1 knows about L3, P?, P?, T?, related to C8".
====

It is reported how many points are awarded to the detective.
The row corresponding to the target clue is excluded from further interrogations.

If all clues of the character are uncovered, a "All clues discovered" token is placed on the character card. Further interrogations of the character are impossible.

[start=6]
6. If "D12 < Threshold", the target clue is not uncovered. Only the relationships are reported:
"Character P1 knows about L3, P?, P?, T?, but did not want to share details."

== Laboratory Expertise

If it is necessary to confirm a hypothesis, detectives can order a laboratory expertise to check the existence of a relationship between *uncovered* entities that may not have direct references to each other.
For example, determine if the ash found in a location matches the cigarettes found on a character.

During the expertise, there are random factors that affect its result.
This influence is modeled by a D12 roll.

The expertise costs a certain resource. The detective may be awarded points for the expertise.

The scenario defines a relationship table *"Laboratory Expertise"*, which defines the relationship between entities.

Expertise Procedure:

1. The detective specifies the detective's focus — a set of IDs of any *uncovered* entities or any types, e.g., C1, P?.
2. Rolls a D12 die to model the influence of random factors.
3. The result of the expertise is determined from the "Entity Expertise" table.
   This table is only accessible via the master program.
4. Upon completion of the expertise, an "Expert Report" card is filled out, indicating the results of the expertise.

=== Relationship Table "Laboratory Expertise"

|===
|Focus           |Threshold|Clue
| P2, L3, L5, T6 |10.0     | C2
| P3, C3, C4     |7.1      | C4
|===

*Focus* — tabular expertise focus (same as <<search-focus, tabular search focus>>).

The detective's focus has a restriction:
*Among the elements of the detective's focus, there must be at least two entity IDs.*

*Threshold* — the threshold of the influence of random factors on the expertise.
A _real_ number that is compared with the number rolled on the D12 die.
If the roll is less than the "Threshold", it is considered that the negative influence of factors is significant, the expertise was conducted poorly, and the target clue is not uncovered.
In the scenario, the influence threshold is set so that the most important clues for the investigation have higher threshold values. The possibility of specifying fractional values allows for "fine" ordering of clues.

To reduce the influence of randomness, low threshold values should be specified; to eliminate randomness — less than 1.

*Clue* — the target clue, essentially a laboratory report detailing the relationship.
For example: "The discovered fingerprints match."

=== Expertise Cost, Awarded Points

*Expertise Cost* — the resource cost of any expertise is the same and is determined in the scenario.
Determines how many resources are spent for conducting an expertise. The detective learns the cost at the beginning of the game.

*Points for Expertise* — the base number of points that may be awarded to a detective for an expertise.
The same base number of points is determined for all expertises in the scenario.
Base points are awarded to the detective for uncovering a clue.
When awarding base points, one additional point is awarded for each _entity_ match in the focus (regardless of whether the clue is uncovered).

== Algorithm for Identifying the Relationship Row in the Expertise Table

The goal of the relationship identification algorithm is: to determine the existence of a relationship and additional data for entities that best match the detective's focus and the D12 roll.

Rows for which the target clue was uncovered in previous expertises are excluded from consideration.

[[expertise-quality]]
1. For each row, the focus weights EQ, EQT, NE are calculated (see <<focus-weights-comp, calculation of focus match weights>>).
2. The row with the maximum "EQ", maximum "EQT", minimum "NE", minimum "Threshold" is found.
3. If "EQ = 0", no relationship is identified. The target clue is not uncovered.
It is reported that no relationship was identified:
   * Unmatched entity IDs and types from the detective's focus.
   * Entity types of unmatched IDs in the tabular focus (as recommendations).
====
For example: "No relationship identified between L2, P4, C?. Expertise recommended for L?, P?, T?."
====

[start=4]
4. If "EQ = 1", relationships for one entity are identified. The target clue is not uncovered. "General" relationships are reported:
   * Matched entity IDs in the detective's focus.
   * Entity types in the tabular focus, except those matching IDs from the detective's focus.

====
For example, for the detective's focus "L2, L3, P?" and the tabular "L2, L4, P1, T3":

"Relationship identified between L2 and L?, P?, T?."
====

[start=5]
5. If "EQ > 1", the relationship is found.
The following are prepared for reporting:

   * Matched entity IDs in the detective's focus.
   * Entity types in the tabular focus, except those matching IDs from the detective's focus.

[start=6]
 6. If "D12 >= Threshold", the influence of factors is low, the target clue is uncovered.

====
Message about the discovered relationship and the uncovering of the clue (expertise report):
   "Relationship identified between L2, P4 and L?, T?, P?. Expertise report C3".
   It is reported how many points are awarded to the detective.
====

The row with the uncovered clue is excluded from further expertises.

If the target clues for all rows of the expertise table have been uncovered, a "Laboratory closed" token is placed. Further expertises are impossible.

[start=7]
7. If D12 < Threshold, the influence of factors is high, the target clue is not uncovered.

====
It is reported: "Relationship identified between L2, P4 and L?, T?, P?. Conducting the expertise was difficult."
====

== Focus Statistics

Obtaining statistics on tabular focuses is an optional mechanic; the possibility of its use is regulated in the scenario.
Obtaining statistics may be allowed to reduce the difficulty of the game for beginners.

Obtaining focus statistics can be either free or cost a certain resource (indicated in the scenario).

Statistics are calculated at the moment of the request — rows for which the target clues have already been uncovered are not taken into account.

[[location-glue-hint]]
=== Location Statistics

Before conducting a search, a detective may request current statistics for a location:

1. The percentage of each entity type in all tabular focuses for the location is calculated.
2. Statistics are provided: "Possible relationships of the location with P 45%, L 30%, T 25%, C 0%."

[[personage-glue-hint]]
=== Character Statistics

Before conducting an interrogation, a detective may request current statistics for a character:

1. The percentage of each entity type in all tabular focuses for the character is calculated.
2. Statistics are provided: "The character may know about P 45%, L 30%, T 25%, C 0%."

[[expertise-glue-hint]]
=== Laboratory Statistics

Before conducting an expertise, a detective may request current statistics for the laboratory:

1. The percentage of each entity type in all tabular expertise focuses is calculated.
2. Statistics are provided: "The laboratory can determine relationships between P 45%, L 30%, T 25%, C 0%."

== Direct Entity Relationships

Direct relationships between any entities are possible through mention in a clue, as well as in the "Entity Expertise" table.
Relationships are also defined according to the table:

|===
| |Location |Time |Clue |Character

|*Location* |Transitions between locations| Clue |Discovery of clue in location| Clue
|*Time* | * |Adjacent ticks|Clue|Clue
|*Clue* | *  | *  |Clue| Discovery of clue on character
|*Character* | * | *  | *  |Clue
|===

== Answers

The "Answers" section is described in the scenario and is revealed at the end of the game.
The section consists of several subsections, each revealing specific aspects of the crime.
Part of the answers is contained in the descriptions of other entities, for example, which locations and passages are hidden, the falsity of clues, event logs, the importance of clues for the investigation, etc.

=== Crime Description

Briefly describes in words the course of the crime, the locations, characters, and clues involved.

=== Recommended Investigation Strategy

The recommended sequence of searches, interrogations, expertises. Which clues to use, which to expect.
